# Open Questions

## Backstock Reports

Resolved direction:

1. Submitted backstock reports should appear inside the existing reports area.
2. Backstock reports should use snapshot semantics.
3. The editable draft can remain local in MVP after generation returns, even though photo upload and generation require backend support.
4. Backstock photos should be optional, and when used they should act as up-front source input for prefill or auto-generation rather than later evidence attachments.

Remaining questions:

1. Does submitting a backstock report update inventory immediately, or create a separate review or approval step?
2. How should low-confidence or partial-coverage counts surface for operator correction?
3. Should submitted detail always retain source photos, or can that become configurable later?

If new product or UX decisions emerge during review, add them here before implementation begins.
